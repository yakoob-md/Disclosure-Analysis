Phase 1: Data Preparation
Tasks:

Download CMU Enron dataset SQL version (Shetty & Adibi schema)
Load into PostgreSQL locally: psql -U postgres -f enron.sql
Deduplicate: SELECT DISTINCT ON (sender, date, SUBSTRING(body, 1, 200)) — MD5 hash dedup preserves forwarded copies
Alias resolution: build alias_map.csv using rapidfuzz.process.extractOne on email prefixes against canonical employee list; threshold = 85
Parse headers with Python email library; extract From, To, CC, BCC, Date, Subject, Body into a clean dataframe
Normalize timestamps to UTC using pytz; add month_index column (0 = Jan 2000)
Filter: remove auto-generated system emails (sender domain ≠ @enron.com, @ect.enron.com, etc.)
Strip quoted reply text: regex r'^(>|From:|Original Message).*$' multiline
Tools: PostgreSQL, Python (pandas, email, pytz, rapidfuzz, hashlib)

Input: Raw Enron SQL dump (~500k emails)

Output: emails_clean.parquet — ~250k emails with resolved aliases, UTC timestamps, clean body text

Challenges: Alias resolution mismatches for non-Enron addresses; partial quoted text removal

Success Criteria: < 5% alias resolution failures; deduplication reduces corpus by 20–35% (expected range from literature)

Phase 2: Labeling Strategy
Tasks:

Define annotation guide (2–3 pages): one page per dimension with 5 positive + 5 negative examples each
Sample 500 emails: stratified by sender role (CEO/VP/Director/Analyst), time period (pre-crisis/crisis), and thread type (single/reply/forward)
GPT-4 labeling: structured JSON prompt, temperature=0, seed=42, model version pinned; store raw API response
Human annotation: yourself + one peer, independently annotate 150/500 emails
Compute Cohen's Kappa per dimension using sklearn.metrics.cohen_kappa_score
Hard gate: if κ < 0.6 on any dimension, revise annotation guide for that dimension and re-annotate
Use GPT-4 labels for remaining corpus; 500-email human-validated set = test split
Tools: OpenAI API (openai >= 1.0), sklearn, Google Sheets (for annotation coordination)

Input: emails_clean.parquet, annotation guide

Output: emails_labeled.parquet with columns: [email_id, body, sender_id, recipients, timestamp, month_index, y_type, y_intent, y_audience, y_risk, label_source]

Challenges: GPT-4 rate limits (~60k tokens/min on Tier 1); κ failures requiring taxonomy revision

6. NETWORK ANALYSIS — HOW METRICS DETECT RISK
Metric	Computation	What it Detects
In-degree centrality	# emails received / (n-1)	Information receivers — potential targets of manipulation or excessive information flow
Out-degree centrality	# emails sent / (n-1)	Information broadcasters — employees spreading information widely; high out-degree + high FIN label = critical risk signal
Betweenness centrality	Fraction of all-pairs shortest paths passing through node	Broker detection — employees who sit between organizational silos; a sudden betweenness spike indicates a new brokerage role, often preceding disclosure events
Closeness centrality	Inverse of average shortest path to all other nodes	Information reach — how quickly can this employee's communications spread through the network? High closeness + inappropriate disclosure = network-wide contamination risk
Δ_betweenness (derivative)	betweenness[t] – betweenness[t-1]	Behavioral anomaly — the single most predictive feature; normal hubs have stable betweenness; criminals cover their tracks by suddenly changing communication patterns
Detection logic example:

IF Δ_betweenness[sender, t] > 2σ
AND email.y_type = FIN
AND email.y_audience = INAPPROPRIATE
AND recipient ∈ ExternalParty
THEN risk_score = HIGH (rule-based escalation, regardless of model prediction)
This rule-based escalation on top of the ML model is itself a publishable contribution — it demonstrates that graph topology carries signal the text classifier cannot capture.

Novel Contribution 1 — Temporal Centrality Derivatives as Behavioral Features
What is new: Computing monthly centrality on evolving graph snapshots and using the rate of change (first derivative) as an ML feature. No published paper does this for disclosure detection on Enron.

Why it matters: Static centrality says "who you are in the network." The derivative says "how your role is changing." Role change precedes behavioral disclosure — this is the pre-crisis behavioral drift signal.

How to implement:

Novel Contribution 2 — Multi-Task Multi-Label Disclosure Taxonomy with Role-Conditioned Audience Scoring
What is new: The four-dimensional label schema itself — no paper has operationalized "disclosure" as a structured 4-head prediction problem on Enron. The audience-appropriateness dimension is specifically novel because it requires the KG (role hierarchy) to compute ground truth.

Why it matters: This reframes the problem from binary anomaly detection (fraud/not-fraud) to structured risk profiling with actionable dimensions. Each dimension corresponds to a compliance intervention.

How to implement: Define audience ground truth via KG query:

cypher
MATCH (sender:Employee)-[:REPORTS_TO*1..3]->(exec)
WITH collect(exec.id) AS authorized_chain
MATCH (e:Email {id: $email_id})-[:RECEIVED]->(r)
WHERE NOT r.id IN authorized_chain AND NOT r:Employee
SET e.y_audience = 'INAPPROPRIATE'
Novel Contribution 3 — LLM Prompt Consistency as a Forensic Reliability Metric

Why it matters: This is Gap 5 in the Gemini doc and has never been studied. Papers that use LLMs for classification never ask "would this verdict change if the question was phrased differently?" In forensic NLP, this is the critical validity question.

9. EVALUATION PLAN
Metrics
Macro-F1: Primary metric. Weighted equally across all classes — critical because your label distribution will be imbalanced (most emails are LOW risk / NONE type). A model that always predicts NONE would have high accuracy but near-zero macro-F1.
Per-label F1: Required for multi-label; shows which disclosure types the model struggles with
Cohen's κ: Required for annotation validity (Phase 2 gate)
Wilcoxon signed-rank test: Required for any claim of model A > model B. Use scipy.stats.wilcoxon(f1_A, f1_B) on per-sample scores.
Evaluating KG Usefulness
The KG contribution must be isolated via ablation:

Ablation 1: Remove graph features from BERT → measure F1 drop
Ablation 2: Replace dynamic centrality with static centrality → measure F1 drop
Ablation 3: Remove role-hierarchy query from audience label computation → measure κ drop in annotation
Each ablation is a publishable finding.

Validating Insights (Beyond Predictions)
Temporal drift curve: Plot betweenness vs. time for top 5 executives. Annotate crisis events. Does the KG show early warning signals?
Role-appropriateness heatmap: Which roles have the highest % INAPPROPRIATE audience scores? (Expected: Traders > Analysts > Executives — this would be a novel finding)
RAGAS faithfulness: For GraphRAG explanations, faithfulness_score = % of explanation claims that are directly supported by the KG path
